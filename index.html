Sublime3的文件名中含中文乱码bug解决

发表于 2016-12-25

首先，sublime文件名中文乱码是个bug，在显示dpi高于100%就会触发，我的thinkpad p50的dpi是125%，所以中招。

解决办法很简单：

  1. 打开 Preferences –> Preference, Settings-User在最后加一句：

“dpi_scale”: 1.0

加好后应该类似这样：

{
    "ignored_packages":
    [
        "Vintage"
    ],
    "dpi_scale": 1.0

}

注意多一个逗号，这是个python的hash-table，保存后重启一下就ok了。

hexo+Github在windows 10下的安装配置

发表于 2016-12-24

在windows 10下折腾hexo的环境来写博客，这次折腾的部分有git/node.js/hexo, hexo是node.js写的，所以得先安装node.js；然后，在windows上部署hexo变更到Github的时候，连接Github不想每次deploy都要输入密码，最好配置好ssh-key，这时候Git for windows里面有在linux下必要的一些小程序，比如ssh-keygen来生成密钥对很方便。不多废话，下面按顺序来。

1. Git安装

下载 Git for windows: 下载地址

直接下载最新版安装即可，很方便的，会自动安装上Git Bash和Git GUI，我们只要使用Git Bash就ok

2. 生成ssh key

生成ssh key是为了以后部署hexo文章到github时候方便用key认证身份，而无需手动输入密码：

安装好Git for windows之后，桌面上有Git Bash,双击打开，或者你也可以直接用cmd或者powershell来输入，安装时自动注册了环境变量，可以直接调用的到

ssh-keygen -t rsa -C "你注册Github的邮箱地址"

中间可以对密钥加密，或者留空，按两次回车都可以
找到刚才生成密钥对的位置, 默认位置是在 C:\users\.ssh
把这个目录下的id_rsa.pub文件内容粘贴出来，然后登陆到你的Github账号，打开如下位置：

右上角，点击你的账号图标 –> Settings –> SSH and GPG keys;
在右侧点击 “New SSH key”, 然后把你的id_rsa.pub公钥内容粘贴到里面，即可。

验证是否可以登录成功：
打开Git Bash，输入

ssh -T git@github.com

如果看到如下字样，恭喜你的ssh key验证通过了：

Hi xxx! You've successfully authenticated, but GitHub does not provide shell access.

3. 安装node.js

node.js下载地址 下载地址

一路下一步回车，结束安装即可。这时候npm工具已经装好，可以安装hexo了。

4. 安装hexo

hexo使用node.js的npm来下载安装的，直接打开cmd或者powershell或者git Bash命令行任意一个，输入如下命令即可安装hexo:

npm install -g hexo

这里提示一个小trick: 国内的网络环境你懂的，速度死慢，如果你有代理服务器，会让你的日子过得舒服一些，npm设置proxy有下面6种方式，优先级由高到低：

- 命令行参数。 --proxy http://server:port即将proxy的值设为http://server:port。
- 环境变量。 以npm_config_为前缀的环境变量将会被认为是npm的配置属性。如设置proxy可以加入这样环境变量npm_config_proxy=http://server:port。
- 用户配置文件。可以通过npm config get userconfig查看文件路径。如果是mac系统的话默认路径就是$HOME/.npmrc。
- 全局配置文件。可以通过npm config get globalconfig查看文件路径。mac系统的默认路径是/usr/local/etc/npmrc。
- 内置配置文件。安装npm的目录下的npmrc文件。
- 默认配置。 npm本身有默认配置参数，如果以上5条都没设置，则npm会使用默认配置参数。

这里我们就用简单的命令行方式：

npm --proxy http://127.0.0.1:1080

ok， 到此hexo就已经安装成功了，接下来是配置hexo让它在windows10上跑起来，并可以自动部署到Github上。

5. 配置hexo

这里附上hexo官方的配置手册，可以文档详细配置，我这里就简单配置下，主要目的是让hexo在windows上跑起来，并且可以在Github上部署，重点是思路。

  • hexo初始化：
    命令行进入想要作为hexo根目录，例如 D:\hexo, 运行下面的命令 hexo init
  • hexo生成静态页面：
    运行 hexo generate 或者简单运行 hexo g
  • hexo本地启动:
    运行 hexo server 或者 hexo s

这时候你应该可以看到命令行窗口已经提示你

INFO  Hexo is running at http://localhost:4000/. Press Ctrl+C to stop

说明，hexo已经在本地的4000端口运行起来了，打开浏览器访问 http://localhost:4000 就可以看到hexo的界面了。

ok，到这里你的hexo环境已经搭建好了，接下来你应该学习一点markdown语法，学着写markdown的文章了。

6. 本地hexo部署到Github

现在hexo环境已经ok，那怎么才能让他在公网环境下跑起来呢？ 这里我用Github提供的Pages服务来部署。
Github Page服务可以直接调用Github里面的Repository，来提供https的免费不限流量的服务。还是很赞的。

前提：

  1. 首先，你得有Github账号，很简单，注册过程不熬述。
  2. 其次，也是很重要的一点就是 要创建一个和你Github用户名名字完全一样的repository才可以直接使用Github Page来匹配你的内容。

这里举例说明下， 如果你的Github 账户名是jm0477，那Github你是可以用 https://github.com/jm0477 来访问，对吧，为了使用Github Pages，你就要创建一个名为jm0477.github.io的repository，以保证你代码部署上来后，用 https://jm0477.github.io 来访问的时候，让Github Pages知道它要去读取的是名字为域名相同的那个repository

当你创建好repository，编辑下本地的hexo根目录下的_config.yml 文件，这是hexo的全局配置文件。
找到deploy: 字段，按照如下格式填好：

deploy:
 type: git
 repo: git@github.com:jm0477/jm0477.github.io.git
branch: master

编辑好保存，然后在命令行下运行：

npm install hexo-deployer-git --save

将deployer保存好，现在就可以使用如下命令来把本地hexo一键部署到github的repository上：

hexo deploy

ok，现在你的hexo已经部署到Github上了，用https://jm0477.github.io/来访问到你的博客站点吧！

PowerShell脚本分享 - 自动更新DNS服务器的RootHint

发表于 2016-07-07

众所周知，全球共有13台DNS根域名解析服务器，没有一个在中国，近些年来亚太地区的根镜像多了很多。

解析的速度也还不错，但是有一些远在北美的，没有镜像的根在国内解析依然有机会会失败，或者超时

今天客户要求我们把服务器的RootHint更新一下，更新完了的确有改善，虽然owner手动解决了此问题，我还是把俩小时的成果共享上来给大家分享吧，提供DNS解析服务的兄弟们可以借鉴，只能用于windows
server的DNS server role

我的服务器环境是Windows Server 2008 R2，用了直接更新文件来达到更新根的目录。

Windows Server 2012有了新的PowerShell cmdlets for DNS server, 实现起来就更方便了，给出命令：

Get-DnsServerRootHint

Set-DnsServerRootHint

剩下的就很简单了。。。。上脚本

#######################################################################
####          Update the RootHint files for all CME DNS servers   #####
####                         Version 1.0                          #####
####                         2016-07-07                           #####
####                         Richie Jing                          #####
#######################################################################

$DNSserverlist          = Get-ADComputer -Filter *  -SearchBase "OU=your_ou,DC=EXAMPLE,DC=GBL"
$CurrentRootHintLocalPath = "c$\Windows\System32\dns\cache.dns" #这里是为了拼凑共享路径来检查文件时间戳和文件内容的
$NewRootHintURL = "https://www.internic.net/domain/named.root"
$NewRootHintLocalPath= "\\server\d$\cache.dns"  #最好是个共享地址方便远程执行


#download the new roothint from https://www.internic.net/domain/named.root
function Download-RootHint()
{
    try 
    {
        Invoke-WebRequest $NewRootHintURL -OutFile $NewRootHintLocalPath
        if(Test-Path -Path $NewRootHintLocalPath)
        {
            return $true
        }
        else
        {
            return $false
        }
    }
    catch
    {
        Write-Warning "RootHint download encountered an error due to $($_.FullyQualifiedErrorId); Reason is: $($_.CategoryInfo.Reason)"
    }

}

#verify single dnsserver roothint files timestamp & compare the roothint file content
function Verify-RootHint($DNSServerName)
{
    $verify_flag=$false
    #verify timestamp
    $timestampCompare = Compare-Object -ReferenceObject (gci $NewRootHintLocalPath).LastWriteTime -DifferenceObject (gci ( "\\"+$DNSServerName+"\"+$CurrentRootHintLocalPath)).LastWriteTime
    if(-not $timestampCompare)
    {
        Write-Debug "Timestamp compare -- no update since last update"
        $verify_flag=$false
    }
    else
    {
        Write-Debug "Timestamp compare -- new timestamp update detected, file needs to be updated"
        $verify_flag=$true
    }

    #verify roothint file content
    $Content_conpare = Compare-Object -ReferenceObject (gc $NewRootHintLocalPath) -DifferenceObject (gc ( "\\"+$DNSServerName+"\"+$CurrentRootHintLocalPath))
    if(-not $Content_conpare)
    {
        #the current roothint file content was the same with the new one
        write-Debug "File content compare -- no update since laste update"
        $verify_flag=$false
    }
    else
    {
        #new content detected, needs to be updated
        write-Debug "File content compare -- new content detected, file needs to be updated"
        $verify_flag=$true
    }
    return $verify_flag
}

#overide the current DNS RootHint file
function Overide-RootHint($DNSServerName)
{
    try{
        Copy-Item $NewRootHintLocalPath -Destination ("\\"+$DNSServerName+"\"+$CurrentRootHintLocalPath) -Force
        return $true
    }
    catch{
         Write-Warning $MyInvocation.InvocationName"ENCOUNTERED AN ERROR Due to $($_.FullyQualifiedErrorId); Reason is: $($_.CategoryInfo.Reason)"
         return $false
    }

}


#Main script part
if((Download-RootHint))
{
    write-host "RootHint file download successfully" -ForegroundColor Green
}
else
{
    write-host "RootHint file download failed,exiting..." -ForegroundColor Red
    exit 1
}

foreach($dns in $DNSserverlist)
{
    if((Verify-RootHint -DNSServerName $dns.name))
    {
        write-host "New change detected, updating..." -ForegroundColor Cyan
        if((Overide-RootHint -DNSServerName $dns.name))
        {
            write-host "RootHint updated for server $dns.name" -ForegroundColor Green
        }
        else
        {
            write-host "RootHint updated failed" -ForegroundColor Red
        }
    }
    else
    {
        write-host "RootHint is already the latest version, ignore...$dns.name" -ForegroundColor Green
    }

}

Nginx + uwsgi 简单部署Django程序

发表于 2016-04-22

最近写了个django的小程序，在本地用manage.py runserver运行的很开心，可是准备放在生产环境的时候就傻咯。

其实部署也是很麻烦的事情，需要考虑django程序如何与web server之间通信，以及django的静态文件如何搞定。

下面记录了刚才俩小时里不断google以及尝试的过程

不要问为啥选择Nginx + uwsgi，
生产环境里有Nginx，觉得Nginx配置起来比较顺手，Djang官方文档推荐配置是Apache+uwsgi，个人觉得小平台，没必要高apache那么臃肿的apache，不喜勿喷。

1. 准备部署环境

首先在部署服务器上用pip安装 Django,uwsgi 以及你编写Django时候依赖的python库:

pip install django

pip install django_wsgi

2. 配置static文件环境

1. 在配置static配置之前，首先要确保你的模板代码中css/js的代码不是写死的绝对路径， 写死的要自己改成django的模板标签 {% load staticfiles %} 然后用{% static "js/jquery-2.0.0.min.js" %} 的方式来调用。

2. 运行python manage.py collectstatic来搜集所有的静态文件，
前提是要在你的project的setting.py里面配置上STATIC_ROOT路径，例如：

STATIC_ROOT = os.path.join(BASE_DIR,”static/“)

3. 配置uwsgi.ini:

#vim /etc/uwsgi.ini

[uwsgi]

vhost = false

plugins = python

socket = 0.0.0.0:3400

master = true

enable-threads = true

workers = 1

wsgi-file = /root/sbi/sbi/wsgi.py

chdir = /root/sbi

processes = 2

max-requests =600

chmod-socket = 664

启动uwsgi测试:

uwsgi –ini /etc/uwsgi/it.uwsgi.ini

仔细看看有没有报错，如果没有报错，就可以继续配置nginx了。

这里需要说明下，如果看到invalid request block size: 21573 (max 4096)…skip 不要惊慌失措。

uwsgi是使用tcp连接来处理请求，并不是http，所以我们前端直接发送的http request会被判定invalid

继续配置nginx，让nginx来代理前端的http请求，并将所有http请求向后转发并使用本地socket与uwsgi通信，就会是正常的

4. 配置Nginx:

#vim /etc/nginx/sites-enabled/your_django_project

server {

listen 8888;

server_name 192.168.100.1;

access_log /var/log/nginx/jastme-access.log ;

error_log /var/log/nginx/jastme-error.log ;

charset utf-8;

location /static {

alias /root/sbi/static/;

}

location / {

uwsgi_pass 127.0.0.1:3400;

include uwsgi_params;

}

}

结束，重启下nginx服务，即可。

通用视图例子:

from django.views.generic import list_detail

publisher_info = {

‘queryset’: Publisher.objects.all()

#lesson1/publisher_info.html 如果没指明，会自动寻找此模板

‘template’: ‘publisher_info.html’

}

urlpatterns = patterns(‘’,

(r’^publisher/$’, list_detail.object_list, {‘publisher_info’: publisher_info})

=================================================================================

模板中调用object_list时，不友好，因为我们不知道object_list里面是什么

我们就可以将publisher_info字典定义一个

‘template_object_name’ : ‘publisher_list’

指定它叫做publisher_list

然后再模板中就可以优雅的调用:

这样就避免了直接使用list_detail下的object_list方法，而改名成更友好的publisher_list直接在模板中调用，很直观

当然有时候一个页面有多个数据源，我们还需要添加额外的处理器

比如说这次处理需要调用Publisher模型及Book模型两种数据

就可以在

publisher_info字典中定义额外：

def find_book():

return Book.objects.all()

publisher_info = {

‘queryset’: Publisher.objects.all()

‘template_object_name’ : ‘publisher_list’

‘template’: ‘publisher_info.html’

‘extra_context’ : {‘book_info’: Book.objects.all()} #定义一个新的处理器

}

但是这样写，有一个小bug， 就是这段程序，只有在启动这个app的时候，在相应的view中运行一次， 以后再次调用，如果数据在此期间修改过。
不会返回更新后的数据.

所以这里要写一个函数引用, 满足通过view渲染的时候才会调用

def find_book():

return Book.objects.all()

publisher_info = {

‘queryset’: Publisher.objects.all()

‘template_object_name’ : ‘publisher_list’

‘template’: ‘publisher_info.html’

‘extra_context’ : {‘book_info’: find_book()} #动态加载Book.bjects.all()

}

函数包装- 来处理复杂的数据过滤

比如说， 想查询制定的publisher下面的所有书

urlpatterns = patterns(‘’,

(r’^books/(\w+)/$’, books_by_publisher’)

)

views.py

from django.shortcuts import get_object_or_404

from lesson1.models import Publisher

def books_by_publisher(request, name):

publisher = get_object_or_404(Publisher, name = name)

return list_detail.object_list(request, queryset =
books.objects.filter(Publisher = publisher),

template_objects_name = ‘book’,

extra_context = {‘publisher’: publisher})

通过 publisher查到指定的publisher, 然后传入queryset来查询指定publisher的书籍，

然后指定了一个友好的template_object_name= ‘book’

然后写了一个额外的context，将publisher和book一并送入list_details.object_list返回
